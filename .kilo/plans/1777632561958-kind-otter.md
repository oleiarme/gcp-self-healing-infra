# Evaluation: gcp-self-healing-infra

## Overall Score: 9.2/10

This is an exceptionally well-engineered project that demonstrates senior-level SRE practices. The roadmap (docs/slo-roadmap.md) accurately targets 9.2 as the realistic ceiling for a single-VM Free Tier deployment.

## What to Add to Reach 10/10

### 1. Testing (Critical Gap)
- **Terraform testing**: Add `terraform test` or Terratest for infrastructure validation
- **Startup script testing**: Unit tests for shell scripts (currently only shellcheck linting)
- **Drift detection**: Automated terraform plan in CI to detect drift

### 2. Multi-Environment Support
- Separate `dev`/`staging`/`prod` workspaces or backends in Terraform
- Currently single environment only — limits safe testing of changes

### 3. Automated Chaos Drills in CI
- `.github/workflows/drill-vm-kill.yml` — run `docs/drills/vm-kill-drill.sh` on schedule
- Compare MTTR against SLO targets automatically
- Currently drill is manual-only

### 4. Cross-Region Backup Replication
- Enable dual-region or multi-region replication for GCS backup bucket
- Current setup: single-region (`US-CENTRAL1`) — zone loss = backup lost

### 5. OS Patching Strategy
- COS auto-updates disabled (pinned image `cos-stable`)
- Add automated OS reimaging workflow or document manual patching procedure in Runbook
- Consider: Google-managed instance groups can auto-update OS images

### 6. Dependency Update Automation (Verify)
- `renovate.json` exists — verify it's actually enabled and configured correctly
- Currently covers Docker images but not GitHub Actions or Terraform providers

### 7. Compliance Documentation
- Add `docs/compliance.md` if targeting SOC2 or similar
- Document change management process (currently relies on PR reviews only)

### 8. Cost Optimization Enhancements
- VM schedule (`schedule-vm-start.yml`, `schedule-vm-stop.yml`) is basic
- Consider: auto-stop on idle detection (no HTTP traffic for N minutes)
- Add cost dashboard showing actual vs budget

### 9. Clarify startup_cos.sh vs startup.sh Usage
- `terraform/main.tf` uses `startup_cos.sh` (COS template)
- But `startup_cos.sh` references Ubuntu-specific paths (`/etc/apt`, `apt-get`)
- Either: fix COS script for actual COS, or switch template to Ubuntu
- Currently: `startup.sh` (Ubuntu) is the working version, COS script is unused/broken

### 10. Incident Communication
- Add status page integration (optional): update external status page on P1 incidents
- Add `docs/incident-communication.md` — what to communicate, when, to whom

## Summary

This is a **senior-level SRE project** with production-grade practices. The 9.2/10 score is accurate per the roadmap — reaching 10/10 requires either:
- Production incident history (to validate SLO assumptions), or
- Multi-region topology (leaves Free Tier, mentioned as out of scope)

**Recommended next steps:**
1. Fix the COS/Ubuntu script mismatch (immediate)
2. Add terraform testing (high ROI)
3. Automate chaos drills in CI (demonstrates operational maturity)
4. Add cross-region backup replication (DR improvement)
